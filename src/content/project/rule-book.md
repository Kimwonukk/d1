# Customer-driven Exception Handling Guide

## 디자인 시스템에서 고객 요청으로 발생하는 예외 대응 가이드

---

## 🎯 목적

이 문서는 다음 문제를 해결하기 위해 작성되었다.

- 고객 요청으로 인해 디자이너가 임의 예외를 생성하는 문제
- 디자인 시스템 규칙 붕괴
- 디자이너가 결정권자가 되는 구조

**목표는 예외를 막는 것이 아니라,  
예외를 ‘요청 유형’으로 흡수하는 것이다.**

---

## 🧠 핵심 원칙

- 고객 요청은 디자인 수정이 아니라 **요구 유형**이다
- 디자이너는 결정자가 아니라 **분류자**다
- 시스템은 고객 요청을 **옵션 / 테마 / 범위 외 작업**으로 나눈다

> 디자이너를 보호하지 못하는 시스템은 오래가지 못한다.

---

## 1️⃣ 고객 요청을 그대로 받지 않는다

### ❌ 잘못된 요청 수용 방식

> “고객이 padding을 더 넓혀달래요”

이 방식의 문제:

- 디자이너가 판단 주체가 됨
- 임시 예외가 누적됨
- 시스템 신뢰도 하락

---

### ✅ 올바른 요청 재정의

> “이 요청은 **브랜드 커스터마이징 요구**입니다”

요청을 **디자인 변경**이 아닌  
**시스템 범위 판단 대상**으로 전환한다.

---

## 2️⃣ 고객 요청 유형 분류표 (필수)

모든 고객 요청은 아래 유형 중 하나로 분류한다.

| 유형 | 설명        | 예시             | 시스템 대응         |
| ---- | ----------- | ---------------- | ------------------- |
| A    | 콘텐츠 변경 | 텍스트 길이 증가 | Auto Layout         |
| B    | 구성 변경   | 카드 개수 추가   | 기존 패턴 사용      |
| C    | 스타일 선택 | 여백을 넓게      | Variant 제공        |
| D    | 브랜드 요구 | 브랜드별 spacing | Theme / Token Set   |
| E    | 일회성 요구 | 이벤트 페이지    | Temporary Exception |

> **값을 바꾸는 요청은 대부분 C 또는 D에 해당한다**

---

## 3️⃣ 디자이너 보호를 위한 공식 문장

디자이너는 아래 문장 중 하나를 사용한다.

- “이 요청은 디자인 수정이 아니라 **시스템 옵션 선택**입니다.”
- “해당 요구는 **기본 디자인 시스템 범위를 벗어나는 커스터마이징**입니다.”
- “이 부분은 **Variant / Theme으로 대응 가능합니다**.”

📌 개인 판단이 아니라 **시스템 언어**를 사용한다.

---

## 4️⃣ Variant로 흡수하는 방식 (C 유형)

### 사용 조건

- 동일 브랜드
- 동일 컴포넌트
- spacing / density 차이

### 예시

```txt
Card Component
 ├─ default
 └─ spacious
```
